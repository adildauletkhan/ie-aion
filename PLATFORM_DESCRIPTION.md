## 1. ОБЩАЯ ХАРАКТЕРИСТИКА ПЛАТФОРМЫ

**Integrated Enterprise** — интегрированная цифровая платформа управления нефтегазовым предприятием, объединяющая геологические, технологические, плановые и операционные данные в единую экосистему с элементами искусственного интеллекта.

### Ключевые характеристики
- **Тип системы:** Корпоративная интегрированная платформа (Digital Twin + ERP-аналитика)
- **Архитектура:** SPA (React/Vite) + REST API (FastAPI/Python) + PostgreSQL
- **Интерфейс:** Двуязычный (RU/EN), тёмная/светлая тема, адаптивный дизайн
- **Мультикомпания:** Переключение между дочерними предприятиями (Workspace Switcher)
- **Доступ:** Авторизация по логину/паролю, JWT-токены, ролевая модель
- **Интеграции:** SAP S/4HANA, SCADA/MES, ручной ввод, расчётные данные

### 3.1 ГЛАВНАЯ СТРАНИЦА (Digital Twin Home)
**URL:** `/`

#### Назначение
Дашборд верхнего уровня — сводные KPI по цепочке создания стоимости + AI-ассистент.

#### Экраны и блоки
| Блок | Содержание |
|------|-----------|
| KPI-карточки (5 шт.) | Мощность добычи, транспорта, переработки, экспорта; общее число активов |
| AI-ассистент | Чат с языковой моделью (GPT-4) по данным платформы |
| Источники RAG | Выбор источников для AI: справочные данные, сценарии, годовые планы, результаты |
| Быстрый запуск сценария | Выбор и запуск расчётного сценария прямо с главной |
| Быстрые действия | Переход к ключевым модулям (анализ, сценарии, планирование, данные) |

#### Объекты данных
- `ExtractionCompany` — добывающие предприятия (capacity)
- `ProcessingPlant` — перерабатывающие заводы (capacity)
- `TransportationSection` — транспортные участки (capacity)
- `ExportDestination` — направления экспорта (capacity)
- `Scenario` — список сценариев для быстрого запуска
- `ChatMessage` — история диалога с AI

#### Функции
- Агрегация мощностей из 4 источников в реальном времени
- RAG-чат: вопрос-ответ по данным платформы через OpenAI API
- Запуск сценария → POST `/scenarios/{id}/run`

### 3.2 ГЕОЛОГИЯ
**URL:** `/geology`

#### Назначение
Геологический модуль для анализа строения месторождений, характеристик пластов, скважин и горизонтов.

#### Вкладки
1. **Месторождения** — список нефтяных полей с фильтрацией
2. **Скважины** — реестр скважин с атрибутами и схемами
3. **Пласты/Горизонты** — характеристики нефтеносных горизонтов
4. **ГИС / Каротаж** — переход к модулю анализа ГИС

#### Объекты данных
- `OilField` — месторождение (id, name, region, type, status)
- `Well` — скважина (id, name, wellType, status, latitude, longitude, depthCurrent, formationName)
- `Horizon` — горизонт (id, name, code, depthTop, depthBottom, lithology, porosity, permeability)
- `WellLog` — кривые ГИС (gamma, resistivity, density, neutron, sonic по глубине)

#### Функции
- Просмотр и фильтрация месторождений и скважин
- Переход к схеме скважины и 3D-модели
- Связь с модулем Модели для 3D-визуализации

### 3.3 МОДЕЛИ (Геологические, Технологические, Интегрированные)
**URL:** `/models`

#### Назначение
Трёхуровневое моделирование месторождения: геология → технология → интеграция.

#### Вкладки верхнего уровня
| Вкладка | Содержание |
|---------|-----------|
| Геология | Горизонты/пласты, Карта пласта, **3D-модель**, Зоны дренирования |
| Технологическая | Скважинное оборудование, Схемы обустройства, Характеристики добычи |
| Интегрированная | Потоки по системе (добыча → транспорт → переработка → экспорт) |

#### Подмодуль: 3D-модель месторождения
Интерактивная Three.js визуализация с профессиональными инструментами:

**Сайдбар объектов:**
- Дерево скважин с типом (добывающая / нагнетательная / наблюдательная)
- Переключатель видимости горизонтов
- Управление разломами, осями, сеткой, изолиниями

**Тулбар управления:**
- Выбор отображаемого свойства: Глубина TVD / Пористость / Проницаемость / Нефтенасыщенность
- Слайдер интервала изолиний
- Time-анимация (0–365 дней) с Play/Pause
- Инструмент измерений (Measure)
- Экспорт скриншота
- Полноэкранный режим

**Функции выделения скважины:**
- Визуальное выделение: 2 пульсирующих кольца + вертикальная стрелка + плавающий ярлык
- Масштабирование выбранной скважины 1.8×
- Желтый буровой ствол (TubeGeometry) с траекторией по глубине и азимуту
- Маркеры глубины на 25/50/75/100% MD

**Панель информации о скважине:**
- Базовые параметры (тип, дебит, обводнённость, давление)
- Параметры траектории: TVD, MD, зенитный угол, азимут, горизонтальное отклонение
- Мини-SVG разрез ствола (отклонение × глубина)

**Цветовая легенда:**
- Вертикальная шкала с градиентом и значениями для каждого свойства

#### Объекты данных 3D-модели
```
WellData {
  id, name, position (THREE.Vector3),
  type: 'production' | 'injection' | 'observation',
  depth, production, pressure, waterCut,
  inclination (°), azimuth (°)
}
ApiHorizon {
  id, name, code, depthTop, depthBottom, lithology, porosity
}
```

### 3.6 ПЛАНИРОВАНИЕ
**URL:** `/planning`

Модуль состоит из 5 вкладок:

---

#### 3.6.1 СЦЕНАРНЫЙ АНАЛИЗ
**Вкладка:** `scenario`

##### Назначение
Анализ влияния производственных сценариев на ключевые показатели компании с привязкой к сохранённым производственным планам.

##### Экраны
1. **Реестр сценарных анализов** — таблица по датам, авторам и компании
2. **Параметры анализа** — выбор базового плана, параметры сценария
3. **Результаты** — сравнение показателей при разных сценариях

##### Сценарные блоки (объединены в единую карточку)
- Базовые сценарии: ограничение добычи, изменение цены, экологические ограничения
- Расширенные сценарии: геополитические риски, технологические изменения, природные катастрофы

##### Объекты данных
```
ScenarioAnalysis {
  id, date, author, company,
  baselinePlanId, scenarioType, parameters,
  results: { production, revenue, cost, npv }
}
SavedPlan { id, name, date, company, data }
```

#### 3.6.2 ПРОИЗВОДСТВЕННЫЙ ПЛАН
**Вкладка:** `production`

##### Назначение
Редактирование, хранение и управление производственной программой предприятия на 5 лет (2026–2030) по 12 разделам.

##### 12 разделов производственной программы
| № | Раздел | Ед. изм. |
|---|--------|---------|
| 1 | Геологоразведочные работы | тыс.тг |
| 2 | Добыча нефти и газа | тыс.т, тыс.тг |
| 3 | Бурение скважин | скв., м |
| 4 | Подземный ремонт скважин | операц. |
| 5 | Экономические показатели | тыс.тг |
| 6 | Техническое обслуживание и ремонт | тыс.тг |
| 7 | Капитальное строительство | тыс.тг |
| 8 | Цифровизация и IT | тыс.тг |
| 9 | НИОКР | тыс.тг |
| 10 | Охрана труда и промышленная безопасность | тыс.тг |
| 11 | Охрана окружающей среды | тыс.тг |
| 12 | МТО и логистика | тыс.тг |

##### Функции модуля
**Редактирование:**
- Inline-редактирование ячеек (объём, цена, сумма) по годам (2026–2030)
- Автоматический расчёт итоговых сумм
- Сохранение версий планов в localStorage

**Список планов:**
- Компактный реестр сохранённых планов
- Загрузка любой версии для редактирования
- Передача планов в Сценарный анализ как базовых линий

**Тулбар действий:**
- **Сохранить** — сохранение текущей версии плана
- **Экспорт Excel** — генерация XLSX с полной структурой, цветовым форматированием (ExcelJS)
- **Печать** — формирование печатной формы производственной программы в HTML
- **Согласование** — открытие панели Approval Workflow

**Validation (Проверка плана):**
- Автоматическая проверка плана по 6 направлениям:
  - Добычные возможности (SAP)
  - Мощность инфраструктуры
  - Запасы нефти по месторождениям
  - Бюджетные ограничения
  - Кадровые ресурсы
  - Регуляторные требования
- AI-анализ (GPT-4o) с финальным заключением
- Отчёт в стиле системы с цветовой индикацией статусов

**Approval Workflow (Согласование):**
- 12 параллельных шагов согласования (по числу разделов)
- Статусы: Ожидание / На согласовании / Согласовано / Отклонено
- Прогресс-бар "Согласовано X/12"
- Кнопки: Отправить на согласование / Сбросить
- Фиксация дат и пользователей согласования

##### Объекты данных
```
ProductionPlan {
  id, name, date, company,
  data: Record<rowId, Record<year, { vol, price, sum }>>
}
ApprovalStep {
  sectionId, sectionName,
  status: 'pending'|'reviewing'|'approved'|'rejected',
  approvedDate, approvedBy
}
```

#### 3.6.3 ПЛАН-ФАКТ АНАЛИЗ
**Вкладка:** `analytics`

##### Назначение
Дашборд сравнения плановых и фактических показателей производственной программы на основе данных из разных источников.

##### Источники данных
| Источник | Тип данных |
|---------|-----------|
| SAP S/4HANA | Финансы, закупки, добыча |
| SCADA/MES | Технологические параметры реального времени |
| Ручной ввод | Корректировки и уточнения |
| Расчётный | Вычисленные показатели |

##### Функции и виды отображения
**Верхняя панель фильтров:**
- Выбор плана (из сохранённых), год (2026–2030), источник данных
- Кнопки: Обновить, Экспорт

**KPI-карточки:**
- Общее исполнение (%), добыча нефти, бурение, капстрой, производственные мероприятия
- Стрелки трендов, прогресс-бары

**Оповещения:**
- Полоса разделов в зоне риска (исполнение < 80%)

**3 вида представления:**

1. **Таблица** — все 12 разделов с план/факт/отклонение/% исполнения + раскрываемые строки
2. **Помесячная динамика** — карточки с мини SVG-диаграммами (план vs факт по месяцам)
3. **По источникам** — агрегация по SAP/SCADA/Ручной/Расчётный

**Цветовая логика (без светофора):**
- ≥ 95% → primary (синий) — отлично
- 80–95% → foreground (нейтральный) — норма
- < 80% → destructive (красный) — отставание

##### Объекты данных
```
SectionSummary {
  id, icon (React.ElementType), titleRu, titleEn, unit,
  plan, fact, source: DataSource,
  monthlyPlan: number[], monthlyFact: number[],
  rows: RowSummary[]
}
RowSummary { nameRu, nameEn, unit, plan, fact, source }
KpiCard { labelRu, labelEn, unit, plan, fact, source, icon }
```

#### 3.7.1 АНАЛИЗ СОБЫТИЙ (Digital Twin / Event Analysis)
**URL:** `/digital-twin`

##### Назначение
Сценарный анализ пропускной способности нефтяной цепочки с расчётом узких мест и оптимизацией.

##### Структура
- **Параметры сценария** — задание входных данных по этапам (добыча, транспорт, переработка, экспорт)
- **KPI карточки** — суммарные мощности по цепочке
- **Таблица дисбалансов** — узкие места и избытки
- **График ограничений** — визуализация пропускной способности
- **Геобурение** (GeoDrillPanel) — расчёт мощности верхнего пласта
- **AI-ассистент** — RAG-чат по данным платформы

##### Функции
- Ручной и расчётный режим задания мощности
- Запуск сценарного расчёта
- Просмотр результатов по этапам цепочки
- Детализация по активам

##### Объекты данных
```
StageInput { stageId, capacity, mode: 'manual'|'calculated' }
ScenarioResult {
  stages: StageResult[],
  bottleneck: string,
  utilization: number,
  recommendations: string[]
}
GeoDrillInputs { porosity, thickness, pressure, viscosity, ... }
```

#### 3.7.2 КРИЗИСНЫЕ СИТУАЦИИ (Crisis Response)
**URL:** `/crisis-response`

##### Назначение
Управление кризисными событиями: регистрация инцидентов, анализ влияния, генерация сценариев ответных мер, исполнение и мониторинг.

##### Жизненный цикл кризисного события
```
Регистрация → Анализ влияния → Генерация сценариев →
Выбор сценария → Исполнение → Мониторинг
```

##### Экраны и вкладки

**Левая панель — Реестр событий:**
- Список кризисных событий с фильтрацией по статусу и типу
- Статусы: open / analyzing / scenarios_ready / executing / resolved / cancelled
- Критичность: low / medium / critical / catastrophic

**Вкладка 1: Регистрация события**
- Форма создания нового события
- AI-генерация описания события
- Категории: equipment_failure, geopolitical, environmental, market, geological, infrastructure
- Затронутые активы с указанием типа (добыча, транспорт, НПЗ, экспорт)

**Вкладка 2: Анализ влияния (Impact Analysis)**
- Оценка влияния на добычу (т/сут), выручку (тыс.тг), риски
- Временные рамки инцидента
- AI-анализ последствий
- Список мер по активам

**Вкладка 3: Сценарии ответных мер**
- AI-генерация сценариев реагирования
- Карточки сценариев: название, описание, шаги, затраты, эффект
- Кнопки: Выбрать / Запустить с индикаторами загрузки

**Вкладка 4: Исполнение (Execution)**
- KPI: статус плана, прогресс (%), финансовое влияние план/факт, разбивка задач
- Прогресс-бар общего исполнения
- Таблица задач по фазам (сгруппировано) с:
  - Статусом (иконки: завершено, в работе, заблокировано, ожидание)
  - Прогресс-баром задачи
  - Ответственным и сроком
  - Подсветкой просроченных задач

**Вкладка 5: Мониторинг (Monitoring)**
- 4 KPI: загрузка НПЗ, объёмы экспорта, финансовое отклонение, активные алерты
- Лента активных алертов по уровням (info / warning / critical)
- Форма записи метрик (ручной ввод)
- Лог мониторинга: история записей с датами, показателями, вариацией

##### WebSocket
- Real-time алерты через WebSocket (`/ws/crisis`)
- Уведомления о новых событиях, изменениях статуса

##### Объекты данных
```
CrisisEvent {
  id (UUID), title, description, eventType, severity,
  status, affectedAssets, startDate, endDate,
  createdAt, updatedAt
}
CrisisScenario {
  id, eventId, name, description,
  steps: string[], estimatedCost, estimatedEffect,
  isSelected, isExecuted
}
CrisisActionItem {
  id, scenarioId, phase, description,
  status, progress, assignee, deadline
}
CrisisExecution {
  id, scenarioId, startedAt, overallProgress,
  actions: CrisisActionItem[]
}
CrisisExecutionMonitoring {
  id, executionId, recordedAt,
  utilizationRefineries, exportVolumes,
  plannedImpact, actualImpact, variancePct,
  alerts: { type, message, severity }[],
  note
}
```

#### Назначение

#### Разделы

## 8. ПРОЦЕССЫ — ОСНОВНЫЕ БИЗНЕС-СЦЕНАРИИ

### Процесс 1: Формирование и согласование производственного плана
```
1. Пользователь открывает модуль Планирование → Производственный план
2. Выбирает компанию в Workspace Switcher
3. Вводит/редактирует данные по 12 разделам на 5 лет
4. Запускает AI-валидацию → получает отчёт о выполнимости
5. Сохраняет версию плана
6. Отправляет на согласование → 12 параллельных шагов
7. После согласования экспортирует в Excel / печатает
8. План становится базовой линией для Сценарного анализа
```

### Процесс 2: Анализ кризисного события
```
1. Оператор регистрирует кризисное событие (тип, критичность, активы)
2. AI генерирует описание и анализ влияния
3. Система генерирует 3–5 сценариев реагирования
4. Руководитель выбирает оптимальный сценарий
5. Запускается исполнение → создаётся план мероприятий
6. Операторы обновляют статус задач в реальном времени
7. Параллельно ведётся мониторинг ключевых показателей
8. По завершении — анализ эффективности принятых мер
```

### Процесс 3: 3D-анализ месторождения
```
1. Пользователь выбирает месторождение
2. Переходит в Модели → Геология → 3D модель
3. Исследует рельеф и геологические слои
4. Выбирает скважину → видит визуальное выделение
5. Анализирует траекторию ствола (глубина, азимут, отклонение)
6. Переключает свойства поверхности (пористость, проницаемость)
7. Использует инструмент измерений для расчёта расстояний
8. Экспортирует скриншот
```

### Процесс 4: Мониторинг план-факт
```
1. Плановик открывает модуль Планирование → План-факт анализ
2. Выбирает сохранённый план и отчётный год
3. Система автоматически подтягивает факт из SAP/SCADA/ручного ввода
4. Видит общее исполнение и разделы в зоне риска
5. Переключает вид: таблица / помесячная динамика / по источникам
6. Экспортирует отчёт по отклонениям
```

## 9. ОЦИФРОВКА ОБЪЕКТОВ И ФОРМИРОВАНИЕ ЦИФРОВОГО ДВОЙНИКА

Этот раздел описывает полный жизненный цикл перевода физических объектов месторождения (схем, скважин, оборудования) в цифровые модели с последующей привязкой к учётным и мониторинговым системам.

---

### 9.1 КОНЦЕПЦИЯ ЦИФРОВОГО ДВОЙНИКА ОБЪЕКТОВ

**Цифровой двойник объекта** — это цифровая копия физического объекта (скважины, установки, месторождения), которая содержит:
- Геометрическое и топологическое описание (схема, конструкция)
- Технические характеристики и паспортные данные
- Привязку к учётным системам (SAP, реестры)
- Real-time телеметрию из SCADA/датчиков
- Историю событий, ремонтов, отказов
- Расчётные и прогнозные характеристики

```
Физический объект
      ↓  оцифровка
Технологическая схема / Конструктивная модель
      ↓  паспортизация
Объект данных с атрибутами (SAP, техпаспорт)
      ↓  подключение мониторинга
Живой цифровой двойник (real-time данные + история)
      ↓  аналитика
Принятие решений (планирование, ТОиР, оптимизация)
```

### 9.2 ОЦИФРОВКА ТЕХНОЛОГИЧЕСКОЙ СХЕМЫ МЕСТОРОЖДЕНИЯ

| Отмена / Повтор | Undo/Redo история действий |
| Экспорт PDF | Выгрузка схемы в PDF |
| Экспорт XML | Выгрузка в формате ISO 15926 / P&ID XML |

**Левая панель — Библиотека объектов (ObjectToolbox):**

Поиск по наименованию или коду объекта в реальном времени.

### 9.3 ОЦИФРОВКА КОНСТРУКЦИИ СКВАЖИНЫ

#### Модуль: WellSchemeEditor
**URL:** `/models` → выбор скважины → вкладка «Конструкция скважины»  
**Технология:** SVG-холст с drag-and-drop по глубине (0–1000 м)

#### Описание процесса

Технолог создаёт вертикальный разрез скважины, размещая компоненты на глубинной шкале. Схема отражает реальную конструкцию с привязкой к точным глубинам спуска.

#### Типы схем по способу подъёма

| Тип | Обозначение | Состав оборудования |
|-----|-------------|-------------------|
| Штанговый глубинный насос | ШГН | Устьевая арматура, кондуктор, обсадные колонны, НКТ, якорь НКТ, штанговый насос, перфорация, штанговая колонна, станок-качалка, сальник |
| Электроцентробежный насос | ЭЦН | Устьевая арматура, кондуктор, обсадные колонны, НКТ, кабель ЭЦН, обратный клапан, газосепаратор, насос ЭЦН, протектор, ПЭД, перфорация, станция управления, трансформатор |

#### Интерфейс редактора скважины

**Левая панель — Палитра компонентов (WellComponentPalette):**

Компоненты сгруппированы по категориям:

| Категория | Компоненты |
|-----------|-----------|
| Обсадные колонны (Casing) | Кондуктор, Технич. колонна, Промеж. колонна, Эксплуат. колонна |
| Насосно-компрессорные трубы (Tubing) | НКТ, Якорь НКТ |
| Цементирование (Cement) | Цементный стакан, Цемент |
| Перфорация | Перфорация |
| Устьевое оборудование (Wellhead) | Устьевая арматура, Сальник, Превентор |
| Оборудование ШГН (SGN) | Штанговый насос, Штанговая колонна, Станок-качалка |
| Оборудование ЭЦН (ECN) | Насос ЭЦН, Газосепаратор, Протектор, ПЭД, Кабель ЭЦН, Обратный клапан, Станция управления, Трансформатор |

**Центральный холст — Вертикальный разрез:**
- Ось глубины: 0–1000 м (масштаб 0.6 px/м)
- Центральная линия ствола скважины
- Компоненты перетаскиваются и фиксируются на нужной глубине
- Перемещение размещённых компонентов drag-and-drop
- Визуальный предпросмотр при перетаскивании (пунктирный блок с меткой глубины)
- Удаление компонента по кнопке
- Цветовое кодирование по типу компонента

**Переключение типа насоса:**
- ШГН ↔ ЭЦН — при переключении загружается стандартная компоновка
- Стандартные компоновки отражают типовое расположение для каждого типа

**3D-превью конструкции скважины:**
- Кнопка «3D-вид» → открывает `WellEquipment3DViewer`
- Three.js визуализация: цилиндрические колонны, компоненты по глубинам
- Возможность повернуть, приблизить, рассмотреть конструкцию

**Правая панель — Аналитика по скважине (WellRightPanel):**

Вкладка 1: **Динамограммы (ШГН)**
- Эталонная динамограмма + актуальная по датам
- Паттерны: нормальная работа, газовое влияние, рабочий ход, недогрузка
- Интерактивный SVG-график (нагрузка кН × ход м)
- Карточки: дата, тип хода, цвет линии

Вкладка 2: **Характеристика насоса ЭЦН (Q-H кривая)**
- Q-H кривая (дебит м³/сут × напор м)
- Несколько кривых по датам замеров
- Отображение рабочей точки
- Интерактивный SVG-график

### 9.5 ПРОЦЕСС ФОРМИРОВАНИЯ ЦИФРОВОГО ДВОЙНИКА

#### Полный жизненный цикл оцифровки

**Этап 1 — Первичная оцифровка (разовая работа)**
```
Источники данных:
  - Технологические регламенты, проектная документация
  - Исполнительная документация на строительство скважин
  - Паспорта оборудования (бумажные → цифровые)
  - P&ID схемы (AutoCAD, AVEVA, SmartPlant)
  - ГИС-данные, топографические карты

Действия:
  1. Создание схемы месторождения в FieldSchemeEditor
  2. Привязка каждого объекта к SAP-коду (sap_asset_code, sap_tech_place)
  3. Заполнение технических паспортов
  4. Создание конструкций скважин в WellSchemeEditor
  5. Привязка к геологическим горизонтам (perforations → horizon_id)
  6. Верификация инженером-куратором
  7. Сохранение базовой версии (Baseline)
```

**Этап 2 — Подключение учётных систем (интеграция)**
```
SAP S/4HANA (PM — Plant Maintenance):
  → sap_asset_code привязывает объект схемы к SAP-активу
  → Синхронизация: статус PM (active / in_maintenance / standby)
  → Синхронизация: история ремонтов, уведомления об отказах
  → Синхронизация: план ТОиР на год

SAP S/4HANA (MM — Materials Management):
  → Список запчастей по объекту
  → Заявки на материалы напрямую через объект схемы

ERP-атрибуты, записываемые в объект схемы:
  - sap_pm_status: string     // 'AVLB' / 'INST' / 'INAC'
  - last_maintenance_date
  - next_maintenance_date
  - maintenance_history: []
```

**Этап 3 — Подключение мониторинговых систем**
```
SCADA / MES (телеметрия в реальном времени):
  → Каждый объект схемы имеет tag-адрес в SCADA
  → Параметры, поступающие онлайн:
     · Скважина:  Q_нефти, Q_жидкости, Q_воды, Q_газа,
                  P_буф, P_затрубное, T_устье, ток ЭЦН, нагрузка ШГН
     · АГЗУ:     Q_суммарный по кусту, давление
     · ДНС/УПН:  загрузка (%), P_вход, P_выход, T_вход, T_выход
     · Насосы:   давление, производительность, вибрация, температура подшипников
     · Трубопрводы: P_вход, P_выход, T, Q

Интеграция через:
  - OPC UA / OPC DA протокол
  - REST API SCADA
  - MQTT-брокер (для IoT-датчиков)
  - File-based (CSV/XML) для legacy SCADA
```

**Этап 4 — Живой цифровой двойник (steady state)**
```
Объект на схеме отображает:
  ✓ Актуальные параметры из SCADA (обновление каждые 5 мин / real-time)
  ✓ Статус из SAP PM (рабочий / в ТОиР / резерв / отказ)
  ✓ Текущий дебит / загрузку
  ✓ Отклонения от норматива (цветовая индикация)
  ✓ Алерты при выходе за пределы (WebSocket → уведомления)
  ✓ Доступ к истории через клик на объект
```

```

        │
        ├──→  Модели → 3D-карта месторождения

        │

        │         Топология схемы = граф для расчёта пропускной способности
        │
        ├──→  Анализ событий (Digital Twin / Event Analysis)
        │         Узлы схемы = объекты для сценарного анализа узких мест
        │
        ├──→  Кризисные ситуации
        │         affectedAssets ссылаются на объекты схемы (sap_asset_code)
        │
        ├──→  Производственный план
        │         Объём добычи распределяется по скважинам и кустам схемы
        │
        └──→  Plan-Fact Analysis
                  Факт отбирается из SCADA по tag-адресам объектов схемы

WellSchemeEditor (конструкция скважины)
        │
        ├──→  3D-модель → траектория ствола
        │         inclination, azimuth из паспорта скважины
        │
        ├──→  Анализ ГИС
        │         Интервалы перфорации → горизонты ГИС
        │
        ├──→  Back Allocation
        │         Дебит скважины → горизонты (по интервалам перфорации)
        │
        └──→  Производственный план
                  Планирование работ (ПРС, КРС) по конкретным скважинам
```

  ├── Размещение скважин на схеме
  ├── Прокладка трубопроводов и добавление объектов (АГЗУ, ДНС, УПН)

  ├── Добавление НКТ, пакеров, насосов, перфорации

Шаг 5. Верификация и приёмка
  ├── Проверка главным технологом / геологом
  ├── Сравнение с бумажными паспортами
  ├── AI-рекомендации по схеме (анализ оптимальности)

Шаг 6. Подключение источников данных
  ├── Маппинг объектов схемы → tag-адреса SCADA
  ├── Настройка синхронизации с SAP PM
  ├── Активация WebSocket-телеметрии

Шаг 7. Эксплуатация цифрового двойника
  ├── Мониторинг показателей в реальном времени
  ├── Автоматическая сигнализация (алерты при отклонениях)
